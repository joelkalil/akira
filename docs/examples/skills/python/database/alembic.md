# Alembic Guidance

## Project Context

- Alembic version: 1.14.0
- Migration path: alembic/versions
- Database: postgres

## Migration Workflow

- Generate migrations from intentional model changes, then review the diff.
- Keep each migration focused on one logical schema change.
- Name revisions with a clear human-readable message.
- Include reversible `downgrade()` logic unless the project explicitly omits it.
- Keep data migrations small, idempotent, and documented.

## Schema Changes

- Use `op.create_table`, `op.add_column`, and related Alembic operations.
- Add indexes and constraints explicitly.
- For non-null columns on existing tables, use a staged migration when data
  backfill is needed.
- Avoid relying on database-specific defaults unless the project targets that
  database intentionally.
- Keep naming conventions consistent with the project's Alembic environment.
- Review autogenerated constraint and index names before committing.

## PostgreSQL Notes

- Use concurrent index creation only when the environment supports autocommit
  migration steps.
- Be careful with enum changes and irreversible type conversions.

## Error Handling

- Catch specific exceptions and handle only failures the code can recover from.
- Preserve exception context with `raise ... from exc` when wrapping errors.
- Avoid broad `except Exception` blocks in request handlers and library code.
- Validate inputs at boundaries and keep domain functions focused on domain work.
- Return actionable error messages without leaking secrets or internal tokens.

## Review Checklist

- Confirm upgrade and downgrade paths match.
- Confirm generated SQL is acceptable for the target database.

## Avoid

- Editing already-applied migrations unless the project is still unreleased.
- Mixing unrelated schema and data changes.
- Dropping columns or tables without an explicit recovery plan.
- Migrations that require network access or cloud credentials.
